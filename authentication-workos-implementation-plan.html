Authentication implementation plan

WorkOS authentication for Nebula Secrets

A production-oriented plan to replace the spoofable development identity selector, bind WorkOS identities to Convex user records, and give System Administrators a safe UI for choosing, validating, and enforcing the authentication provider.

Initial provider: WorkOS AuthKit Frontend: React + Vite Backend: Convex Prepared 21 July 2026 Status: proposed
01 · Outcome

What this implementation should deliver

WorkOS becomes the only trusted identity source. Convex derives the caller from the verified JWT, while Nebula keeps ownership of application roles, suspension status, environment grants, and browser-held encryption keys.

A

Authenticated sessions

Users sign in through WorkOS AuthKit. The Vite client forwards a WorkOS access token through the Convex auth provider, and protected functions reject unauthenticated requests.

B

Trusted authorization

Every Convex function resolves the current user from ctx.auth.getUserIdentity(). Client-supplied actorUserId values disappear from the public API.

C

System Admin control

A dedicated Authentication page lets System Administrators stage WorkOS, review readiness, define account admission policy, and enforce sign-in without exposing provider secrets.

✓
Recommended scope boundary

The initial release supports one WorkOS configuration per deployment and invitation-only account linking. It does not implement multi-tenant provider configurations, WorkOS Directory Sync, organization switching, or additional providers.

02 · Current state

Repository findings that shape the plan

Authentication is not an isolated login-screen change. The selected user ID currently flows through nearly every query and mutation, so the secure migration must replace that authorization contract end to end.

Identity and access today

  • src/App.tsx persists nebula-selected-user in local storage and exposes a switchable identity selector.
  • Public Convex functions accept actorUserId and trust it after a database role check.
  • The user schema has only developer and admin roles, with no external identity field.
  • The first bootstrap user is named “System Administrator” but is stored with the admin role.

Provider integration today

  • src/main.tsx uses plain ConvexProvider, so authenticated tokens are not sent.
  • convex/auth.config.ts and root convex.json do not exist.
  • No WorkOS packages are installed.
  • Local development is fixed at 127.0.0.1:5173; callback, CORS, and homepage values must use that exact origin.
!
Security consequence

Adding AuthKit to the page without removing actorUserId authorization would leave the core spoofing vulnerability intact. Backend identity derivation is the release gate.

03 · Architecture

Provider configuration has two layers

The safest design separates deployment credentials from runtime policy. System Administrators own the in-app policy and activation workflow; infrastructure credentials remain in the secret stores that already control the deployed frontend and Convex backend.

Deployment adapter operator-managed

  • WorkOS Client ID and API key
  • Callback, homepage, and CORS origins
  • convex/auth.config.ts JWT validation
  • Vercel and Convex environment variables
  • Dev, preview, and production isolation
⇄

Application policy System Admin-managed

  • Selected provider: WorkOS
  • Draft, verified, or enforced state
  • Invitation-only admission policy
  • Allowed email domains, if enabled
  • Readiness checks and audit history
🔒
Secret handling rule

The Authentication UI must never request or persist WORKOS_API_KEY, deploy keys, cookie secrets, or access tokens. It may display public values, non-sensitive fingerprints, expected callback URLs, and links/instructions for the deployment operator.

Request path after enforcement

1 · Sign inWorkOS hosted AuthKitAuthenticates the person and returns to the configured callback.
2 · ClientAuthKitProviderOwns browser auth state and obtains an access token.
3 · TransportConvexProviderWithAuthKitAttaches and refreshes the token for Convex requests.
4 · VerifyConvex auth configValidates WorkOS issuer, keys, algorithm, and audience.
5 · AuthorizeNebula access helpersMaps tokenIdentifier to a user, then checks status, role, and grants.
04 · Roles

Add a real System Administrator role

Provider configuration is a control-plane permission. It should not be automatically granted to every vault Admin. The proposed hierarchy makes systemAdministrator a superset of admin while keeping environment grants independent.

The last active System Administrator cannot be suspended or demoted. Existing first-bootstrap Admin is promoted during migration; other Admins remain Admin unless explicitly promoted.

06 · Data & APIs

Schema and authorization contract

Nebula already needs first-class user documents for roles, grants, keys, and audit references. WorkOS identity fields should therefore be added to the existing users table rather than creating a parallel user model.

users additions

  • role gains systemAdministrator.
  • authProvider?: "workos".
  • tokenIdentifier?: string, indexed as by_tokenIdentifier.
  • providerUserId? for support/audit display only; never the global authorization key.
  • emailVerified?, identityLinkedAt?, and lastAuthenticatedAt?.

New authConfiguration singleton

  • provider: "workos".
  • state: "staged" | "verified" | "enforced".
  • provisioningMode: "invitationOnly" initially.
  • Optional allowed domains and public configuration fingerprint.
  • updatedBy, updatedAt, enforcedAt.
  • No API keys, deploy keys, access tokens, or cookie secrets.

Backend helper contract

Helper / functionResponsibilitySecurity rule
getCurrentUser(ctx)Read WorkOS identity and resolve the indexed Nebula user.Use canonical identity.tokenIdentifier; return null for no match.
requireActor(ctx)Require an authenticated, linked, active user.No caller-provided user ID.
requireAdmin(ctx)Allow Admin or System Administrator.Role is read from the linked Nebula user record.
requireSystemAdministrator(ctx)Guard authentication configuration and privileged recovery.Reject ordinary Admins.
users.linkCurrentIdentityAttach the verified WorkOS identity to a pre-created user.Exact normalized verified-email match, uniqueness check, and bootstrap restrictions.
auth.statusReturn minimal public sign-in state and authenticated readiness details.Never return secrets or full provider diagnostics to anonymous callers.
✓
Identity-linking rule

Unknown WorkOS users are not auto-promoted or auto-created with vault access. A System Administrator pre-creates the Nebula user by email; first sign-in links only when WorkOS asserts that same verified email and the tokenIdentifier is not already linked.

07 · Delivery

Phased implementation plan

The sequence deliberately keeps the application deployable while its schema and request contract change. Authentication is not declared complete until the legacy actor path is gone.

Phase0

Confirm rollout decisions

Lock the operating model before code changes.

Review gate
  • Choose Convex-managed WorkOS team (recommended) or an existing WorkOS team.
  • Confirm production-ready rollout, including Vercel preview and production.
  • Approve the new System Administrator role and role hierarchy.
  • Confirm invitation-only account admission and exact verified-email linking.
Phase1

Add the WorkOS deployment adapter

Make WorkOS capable of issuing tokens that Convex can verify.

Foundation
  • Install @workos-inc/authkit-react and @convex-dev/workos.
  • Create convex.json with dev, preview, and prod AuthKit settings.
  • Use http://127.0.0.1:5173/callback for this repository’s actual local origin.
  • Create convex/auth.config.ts with the current WorkOS JWT providers.
  • Separate WorkOS staging and production credentials.
  • Add typed client env declarations and fail-fast configuration diagnostics.
Phase2

Widen the schema and migrate roles

Prepare existing documents for authenticated identity mapping without downtime.

Deploy 1
  • Add optional identity fields, the token index, and the new configuration table.
  • Extend role validators to accept systemAdministrator.
  • Update authorization code to treat System Administrator as an Admin superset.
  • Promote the original bootstrap Admin through a bounded migration.
  • Verify there is at least one active System Administrator.
  • Keep new identity fields optional until linking and verification finish.
Phase3

Replace the backend trust boundary

Move every authorization decision from client arguments to verified session identity.

Security critical
  • Implement current-user, Admin, System Administrator, and environment-access helpers.
  • Add identity-linking and current-user functions.
  • Remove actorUserId from access, audit, attachment, project, secret, user, and devtool APIs.
  • Populate actor, owner, updater, and audit IDs from the resolved user.
  • Keep target user IDs only where an administrator acts on another user.
  • Deny valid WorkOS sessions for suspended, unlinked, or unknown users.
Phase4

Build the authenticated app shell

Replace development identity selection with real sign-in, sign-out, callback, and session states.

Frontend
  • Wrap React with AuthKitProvider and ConvexProviderWithAuthKit.
  • Render loading, unauthenticated, linking, suspended, and authenticated states explicitly.
  • Add hosted sign-in and sign-out actions plus /login and callback handling.
  • Remove the identity selector, selected-user local storage, and development warning after enforcement.
  • Keep the device-key gate, keyed by the resolved Nebula user ID.
  • Show the signed-in WorkOS profile without treating it as authorization state.
Phase5

Add System Administration → Authentication

Give the privileged role a guided configuration and enforcement surface.

Admin UX
  • Add provider summary, WorkOS configuration, admission policy, and health sections.
  • Provide managed-team and existing-team setup instructions.
  • Show exact per-environment redirect, homepage, and CORS values.
  • Run backend token/current-user and client-origin readiness checks.
  • Require explicit confirmation before enforcement.
  • Audit configuration edits, verification, enablement, disablement, and identity links.
Phase6

Verify, enforce, and narrow

Complete the migration only after real WorkOS sign-in succeeds end to end.

Release gate
  • Run unit, Convex authorization, migration, UI, and browser flow tests.
  • Link and verify at least two users, including one System Administrator.
  • Exercise suspension, grant revocation, sign-out, refresh, and missing-device-key paths.
  • Enable WorkOS enforcement in staging, then production.
  • Remove all legacy identity code and temporary dual-read logic.
  • Narrow required identity fields only when every active user is linked.

09 · Verification

Acceptance criteria and test matrix

The security property to prove is simple: changing a user ID in a request can no longer change the caller, because public protected APIs do not accept an actor ID.

AuthenticationUnauthenticated calls fail; signed-in WorkOS users reach Convex with a validated identity.
Identity mappingVerified, pre-created users link once; duplicate token or email links are rejected.
AuthorizationRole × environment × ownership combinations pass or fail exactly as designed.
System AdministrationOnly System Administrators can read privileged diagnostics or change auth policy.
SuspensionA valid WorkOS session for a suspended user cannot read keys, secrets, or admin data.
Encryption continuityExisting device keys and envelopes remain bound to the same Convex user document IDs.
Provider configurationClient ID, callback, issuer/JWKS, and environment isolation are validated before enforcement.
AuditabilityIdentity links, auth policy changes, enablement, denials, and role changes create privacy-safe events.
Session lifecycleLogin, callback, refresh, sign-out, expired session, and revoked access behave predictably.
Responsive UISign-in, account-state, and System Administration pages remain usable at 320px and desktop widths.
✓
Release definition of done

npm test, npm run lint, npm run build, and npx convex dev --once pass; a real WorkOS staging login returns to /callback; useConvexAuth() becomes authenticated; and a protected Convex query resolves the expected Nebula user.

10 · Rollout

Safe activation and rollback

Use separate WorkOS staging and production environments. The application should never silently fall back to the legacy selector when an enforced deployment is misconfigured.

Activation sequence

  1. Deploy schema widening and WorkOS provider wiring to development.
  2. Promote and link the first System Administrator.
  3. Verify a second non-privileged account and all device-key paths.
  4. Configure and test a Vercel preview deployment against WorkOS staging.
  5. Provision distinct production credentials and HTTPS callback origins.
  6. Enable enforcement, monitor denials and callback failures, then remove legacy code.

Rollback strategy

  1. Before final narrowing, roll back application enforcement without deleting identity links.
  2. Keep the old role value readable during the migration window.
  3. Do not delete auth audit events or provider fingerprints.
  4. Never expose a production fallback that accepts actorUserId.
  5. If provider access fails after enforcement, show a service-unavailable/auth-configuration screen and use the documented operator recovery process.

Metric

Sign-in success/failure and callback validation failures, without logging tokens or secret material.

Metric

Unknown, unlinked, and suspended-user denials, grouped by non-sensitive reason.

Metric

System Administrator count and provider-readiness state, with alerts for zero active privileged users.
