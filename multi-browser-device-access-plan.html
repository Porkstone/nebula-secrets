Nebula Secrets
Design proposal · Review only
Secure device enrollment

Multi-browser access without sharing private keys.

A design for letting one authenticated user access Nebula Secrets from multiple browsers while preserving client-side encryption and keeping plaintext keys away from Convex.

Recommended direction

Give every browser its own non-exportable key pair. An existing approved browser transfers access by locally re-wrapping environment keys for the new browser.

01

The problem to solve

Authentication and encryption answer different questions.

WorkOS proves identity

A user can already authenticate in another browser. That proves who they are, but it does not give the browser access to encryption keys.

The browser holds the decryption capability

Nebula currently stores a single public key per user. The matching non-exportable private key stays in one browser's IndexedDB and cannot safely be copied.

Design principle: signing in on a new browser must not automatically reveal secrets. Authentication establishes identity; device enrollment grants cryptographic access.
02

Recommended enrollment flow

The existing browser acts as the trusted bridge to the new browser.

1

Sign in and request

The user signs in through WorkOS. The new browser creates its own non-exportable key pair and registers a pending device with its public key.

2

Review on trusted device

An existing active browser displays the request, device details, expiry time, and a short verification code.

3

Re-wrap locally

After approval, the trusted browser decrypts each environment key locally and wraps it to the new browser's public key.

4

Activate

Convex stores the new encrypted envelopes and marks the device active. The new browser can now decrypt the same authorised environments.

No private-key export: the original browser's private key remains non-exportable. Only environment keys are transferred, and only after being re-encrypted for the new device.

04

Device management experience

Users manage their own trusted browsers; administrators manage policy rather than plaintext access.

Pending requests

Review device name, approximate browser details, request time, expiry, and verification code. Approve only from an active device.

Active devices

Rename devices and view approval source, creation time, and last use. Clearly identify the current browser.

Revoked devices

Retain minimal audit metadata while removing access to key envelopes and preventing future device-authorised operations.

ActorPermitted operationsRestrictions

05

When the only browser is lost

Recovery is an explicit product and security decision.

Phase 1: no Local recovery

An Admin can re-grant shared environment keys to a replacement device. Local secrets remain inaccessible because only the lost browser held the necessary private capability.

Optional phase: recovery phrase

At enrollment, generate a high-entropy recovery phrase that encrypts a backup of the user's environment keys. The server stores only encrypted recovery material.

Recovery changes the threat model: a recovery phrase makes access more resilient but creates another credential that can unlock secrets. It should be optional, rate-limited, audited, and protected by WorkOS re-authentication.
06

Revocation and key rotation

Revoking server access is useful, but it cannot erase data already seen by a browser.

ControlEffectLimitation
Revoke deviceReject future device operations and remove its stored key envelopes.Cannot erase plaintext or keys previously cached by that browser.
End WorkOS sessionsStops authenticated API access for compromised sessions.Does not invalidate material already decrypted offline.
Rotate environment keyCreates a new key version and new envelopes for remaining devices.Existing secret payloads require re-encryption for complete cryptographic revocation.

Recommended initial scope: revoke device access immediately and use key rotation as a later, higher-assurance feature.

07

Safe migration of existing users

Use a widen–migrate–narrow rollout so current browsers keep working throughout.

Deploy 1 · Widen

Add the device model

Add device tables and optional references. New code reads both legacy user envelopes and device envelopes, while new enrollments use the device model.

Backfill

Create each legacy device

Create one active device from every existing publicKeyJwk and copy the user's current environment envelopes to that device without changing ciphertext.

Client claim

Associate IndexedDB keys

On the existing browser's next login, associate its legacy private-key reference with the new device record. The non-exportable key itself remains untouched.

Verify

Check coverage

Confirm every active user with a legacy key has an active device and equivalent environment envelopes. Keep migration progress bounded and resumable.

Deploy 2 · Narrow

Retire the single-device path

Switch reads entirely to device envelopes and deprecate legacy fields. Physical field removal can wait until the new model is proven stable.

08

Security controls

Controls that should accompany device enrollment.

Approval integrity

Use a short-lived request, single-use nonce, and optional device signing key. The existing device signs the approval and the server verifies the registered public key.

Step-up authentication

Require recent WorkOS authentication or MFA before approving, recovering, or revoking a device.

Notifications and audit

Record request, approval, rejection, envelope creation, and revocation events without logging key material. Notify the user of device changes.

Abuse controls

Limit pending requests per user, expire them quickly, rate-limit retries, and never reveal whether another email address has a Nebula account.

Decisions for review

Items to agree before implementation begins.

10

Recommended delivery boundary

A focused first release that adds meaningful multi-browser access without weakening encryption.

First release: per-browser device records, pending enrollment, approval from an existing device, locally re-wrapped Local and shared environment keys, device management, revocation, audit events, and the legacy-device migration.
Later release: recovery phrases, full environment-key rotation, automated secret re-encryption, organisation-wide device policy, and advanced risk signals.
